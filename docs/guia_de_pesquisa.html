Qualidade de Código Gerado por Agentes de IA

Um estudo empírico em larga escala usando o dataset AIDev

Guia de pesquisa de mestrado · Autora: Ruth Maria · Atualizado em 22 de junho de 2026 · Repositório: github.com/RuthMaria/aidev-analysis

Resumo

Agentes autônomos de programação (OpenAI Codex, Devin, GitHub Copilot, Cursor e Claude Code) passaram a abrir pull requests (PRs) em projetos reais de código aberto, atuando como "colegas de equipe" de desenvolvedores humanos — fenômeno que a literatura chama de Engenharia de Software (SE) 3.0. Esta pesquisa investiga a qualidade do código produzido por esses agentes, usando o dataset AIDev (~932 mil PRs em mais de 116 mil repositórios). Como o dataset não contém uma métrica direta de qualidade, adotamos proxies observáveis — taxa de aceitação (merge), taxa de rejeição, esforço de revisão — e comparamos os agentes entre si, contra um baseline humano, controlando pelo tipo de tarefa.

Principal achado preliminar

A taxa de aceitação varia muito entre agentes (de 59% a 88%). O OpenAI Codex supera inclusive os humanos em aceitação, enquanto Claude Code empata com humanos (76,8%) e Copilot/Devin ficam abaixo. A qualidade percebida cai sistematicamente em tarefas mais complexas (feat, fix) comparada a tarefas simples (docs, build). Uma análise estática preliminar (RQ5) mostra ainda que a aceitação não acompanha a limpeza do código: o agente menos aceito (Copilot) é o que gera a menor densidade de warnings de lint.

1. Contexto e motivação

A engenharia de software vive uma transição. Após a SE 1.0 (processos manuais) e a SE 2.0 (apoio de IA como autocompletar), a SE 3.0 caracteriza-se por agentes de IA que executam tarefas de ponta a ponta: leem uma issue, escrevem o código, abrem o PR e respondem à revisão. Diferente de assistentes que apenas sugerem trechos, esses agentes contribuem como autores de PRs.

Isso levanta uma pergunta central para a indústria e a academia: o código gerado por agentes de IA tem qualidade suficiente para ser aceito em projetos reais? Responder a isso de forma empírica — e não anedótica — exige dados em larga escala de contribuições reais de agentes, que é justamente o que o dataset AIDev fornece.

Por que isso importa para a tese

Entender a qualidade dos PRs de agentes ajuda a (a) calibrar a confiança que mantenedores devem depositar em contribuições automatizadas, (b) identificar em quais tipos de tarefa os agentes são mais/menos confiáveis e (c) apontar lacunas de qualidade que ferramentas futuras precisam endereçar.

2. Objetivo e questões de pesquisa

Objetivo geral: avaliar empiricamente a qualidade do código produzido por agentes autônomos de IA em pull requests de projetos open-source, comparando-os entre si e com desenvolvedores humanos.

As questões de pesquisa (RQs) que guiam o estudo:

RQ1 Aceitação. Qual a taxa de merge (e de rejeição) dos PRs de cada agente? PRs aceitos são um proxy de qualidade percebida pelos mantenedores.

RQ2 Esforço de revisão. Quanto retrabalho um PR de agente exige antes de ser aceito (pedidos de mudança e comentários de revisão)?

RQ3 Comparação com humanos. Como os agentes se posicionam frente a um baseline de PRs humanos nas mesmas métricas?

RQ4 Efeito do tipo de tarefa. A qualidade percebida varia conforme a natureza da tarefa (feat, fix, docs…)?

RQ5 Qualidade estática intrínseca. Independentemente da aceitação, o código submetido pelos agentes apresenta maior densidade de problemas estáticos (code smells, complexidade, violações de estilo) do que o esperado? Como isso varia por agente? Enquanto RQ1–RQ4 medem a qualidade percebida (aceitação/revisão), a RQ5 olha o próprio código, fechando a lacuna que a literatura aponta em estudos baseados só em taxa de merge.

RQ6 Qualidade dinâmica (testes). O código que os agentes adicionam é bem testado? Medimos a cobertura das linhas adicionadas e o mutation score dos arquivos modificados. Se a RQ5 é a análise estática (sem executar), a RQ6 é a dinâmica (executando a suíte) — e a mutação revela algo que a cobertura sozinha esconde: se os testes realmente pegam bugs, não só se passam pelo código.

RQ5 e RQ6 são extensões exploratórias

As RQs 1–4 refletem o que o dataset responde com alta confiança. A RQ5 (estática) e a RQ6 (dinâmica) são mais exploratórias: dependem dos patches de pr_commit_details, de cobertura parcial (a API do GitHub omite diffs grandes — ver §7). A RQ5 roda sobre uma amostra de arquivos Python; a RQ6 exige executar a suíte e por isso roda sobre uma amostra ainda menor de repos reconstruídos. Ambas são prova de conceito do método.

5. Metodologia

5.1 O problema da definição de "qualidade"

O dataset não traz uma coluna de qualidade. Qualidade de software é um construto multidimensional (corretude, manutenibilidade, legibilidade…). Em um estudo de mineração de repositórios, ela é operacionalizada por proxies observáveis — sinais registrados pelo processo de contribuição que se correlacionam com qualidade.

RQ5: dos proxies de aceitação aos proxies intrínsecos

Os quatro primeiros proxies são extrínsecos (dependem do julgamento de mantenedores). A RQ5 acrescenta proxies intrínsecos, medidos diretamente sobre o código com análise estática: a densidade de warnings de lint e a complexidade ciclomática. A normalização por linhas adicionadas é essencial para comparar PRs de tamanhos diferentes — uma ressalva metodológica recorrente na literatura.

Por que a taxa de merge é o proxy principal

Um PR mesclado significa que mantenedores humanos — que conhecem o projeto — julgaram o código bom o bastante para integrar à base. É o sinal de qualidade mais direto e menos ruidoso disponível no dataset, embora meça aceitação, não corretude absoluta (ver §7).

5.3 Procedimento de análise

  1. Derivar merged e rejected de cada PR a partir das datas.
  2. Agregar por agente para RQ1 e RQ2.
  3. Concatenar PRs humanos (rótulo Human) para RQ3.
  4. Juntar (join) com pr_task_type por id para RQ4.
  5. Para o esforço de revisão, ligar revisões e comentários ao PR via pr_id / pull_request_review_id.

A análise é totalmente reproduzível pelo script quality_analysis.py (ver §9), escrito em Python com pandas.

Para todos os agentes, tarefas de docs/build têm aceitação mais alta que feat/fix/refactor. Ou seja: quanto mais o código mexe na lógica do sistema, menor a taxa de aceitação — evidência de que a qualidade dos agentes ainda é mais frágil em tarefas complexas.

Ele também requalifica a RQ1: parte da liderança do Codex se estreita quando olhamos só tarefas que mexem na lógica do sistema, sustentando a tese de uma lacuna de qualidade em tarefas complexas.

Achado surpreendente: aceitação ≠ qualidade estática

A ordem da RQ5 não reproduz a da aceitação (RQ1). O Copilot, que tem a menor taxa de merge (59%), escreve o código com a menor densidade de warnings; o Cursor concentra mais avisos. Como ler estes números (ressalvas)

São preliminares: (a) amostra de ~1,8 mil arquivos, com poucos do Claude Code (113); (b) só Python; (c) analisamos as linhas adicionadas do diff, não o arquivo completo (limitação do dataset, §7); (d) a regra ALL do ruff é severa e infla a contagem absoluta — o que importa aqui é a comparação relativa entre agentes. A taxa de "% compila" de 100% reflete que, nesta amostra, os fragmentos eram em sua maioria arquivos novos (completos).

6.6 RQ6 O código dos agentes é bem testado? — cobertura e mutação (metodologia)

A RQ5 mede qualidade sem executar o código (análise estática). A RQ6 acrescenta a dimensão dinâmica: executa a suíte de testes do projeto e pergunta quão bem-testado é o código que o agente adicionou. Usamos duas técnicas clássicas de teste de software:

TécnicaO que medeFerramenta
Cobertura de linha/ramoquanto das linhas adicionadas pelo agente é de fato exercitado pela suíte (patch coverage)pytest-cov
Teste de mutaçãoinjeta defeitos (mutantes) no código do agente e mede quantos a suíte mata (mutation score)mutmut

Por que cobertura e mutação? (o ponto central da disciplina de testes)

Cobertura alta não significa teste eficaz. Um arquivo pode ter 100% de cobertura e mutation score baixo: os testes executam o código, mas não verificam o resultado (oráculos fracos). A cobertura diz "o teste passou por aqui?"; a mutação diz "o teste pegaria um bug aqui?". Por isso as duas se complementam — e a mutação é a evidência mais forte de que os testes realmente protegem o código.

Por que medir o código do agente (e não os testes dele)?

Havia duas opções: (A) avaliar a qualidade dos testes que o agente escreve, ou (B) avaliar quão bem-testado é o código que o agente escreve. Escolhemos a (B) porque ela liga diretamente "testes" ao objeto da tese — a qualidade do código gerado. Aplicamos cobertura às linhas que o agente adicionou e mutação aos arquivos que ele modificou.

Motivação (medição direta, sobre todo o subset). Antes da análise dinâmica, uma observação barata sobre pr_commit_details: os agentes já acompanham o código com testes em proporções bem diferentes — de 50% (Claude Code) e 48,6% (Codex) a 28% (Cursor) dos PRs. Mas presença de teste não garante eficácia: é exatamente o que cobertura e mutação vão verificar.

Por que uma amostra curada (e não o dataset inteiro)?

Cobertura e mutação exigem executar a suíte em um repositório reconstruído. O AIDev fornece apenas os patches, não os repos montados; clonar, instalar dependências e rodar testes é pesado e instável (cada repo tem seu setup), e a mutação roda a suíte uma vez por mutante (lento). Logo, rodar em 900 mil PRs é inviável. A RQ6 trabalha sobre uma amostra estratificada por agente — e a própria curadoria/automação vira parte do método (reprodutibilidade por amostra).

Critérios da amostra (e o porquê de cada um), gerada por src/sample_for_testing.py:

CritérioPor quê

Pipeline de execução (passo 2, src/mutation_coverage.py), por PR amostrado:

git clone <repo_url> && git checkout <head_sha>   # reconstrói o estado do PR
pip install -e .                                    # instala o projeto e suas deps
pytest --cov --cov-report=xml                       # roda a suíte com cobertura
# cobertura SÓ das linhas adicionadas pelo agente (patch coverage)
mutmut run --paths-to-mutate <arquivos_.py_do_diff> # mutação só no código do agente

A mutação é restrita aos arquivos do diff (não ao projeto todo) justamente para ser viável — senão o número de mutantes explodiria.

Status e o que será reportado

A amostra já foi gerada (25 PRs, 5 por agente, sorteados de 1.267 candidatos). Os resultados de cobertura e mutation score dependem de rodar o runner (execução pesada, na máquina local). Métricas finais, por agente: % das linhas adicionadas cobertas e mutation score médio dos arquivos modificados. Espera-se que parte dos repos falhe ao instalar/rodar — a tese reporta sobre o subconjunto que rodou (limitação inerente a estudos dinâmicos de mineração).

Síntese: quatro níveis de evidência

Juntando todas as RQs: (1) há uma hierarquia de aceitação estável (RQ1–RQ3), com Codex e Claude Code no topo, já no nível humano; (2) essa vantagem encolhe em tarefas complexas (RQ4), o achado mais robusto; (3) aceitação não equivale a qualidade estática de código (RQ5); e (4) a RQ6 fecha o arco indo da qualidade estática à dinâmica (cobertura + mutação) — perguntando não só se o código é limpo, mas se ele está de fato protegido por testes eficazes. A mensagem para mantenedores e para a tese: confiar em contribuições de agentes exige olhar além da taxa de merge — controlando o tipo de tarefa, inspecionando o código e testando-o.

7. Ameaças à validade

Validade de construto

A taxa de merge mede aceitação percebida, não qualidade intrínseca. Projetos diferem em rigor de revisão; alguns podem aceitar PRs de bots quase automaticamente. A RQ5 mitiga isso triangulando com análise estática do próprio código (§6.5) — mas a RQ5 também tem limites de construto: densidade de warnings de lint e complexidade ciclomática são sinais de superfície, não de corretude; um código pode estar correto e cheio de avisos de estilo, ou "limpo" e logicamente errado. Além disso, a regra ALL do ruff infla as contagens absolutas, então só a comparação relativa entre agentes é válida.

Validade interna

Os volumes são desbalanceados (814 mil PRs do Codex vs. 5 mil do Claude Code), e a composição de tarefas difere entre agentes. Comparações diretas de médias podem enganar; por isso o controle por tipo de tarefa (RQ4) e, idealmente, testes estatísticos e modelos de regressão são recomendados.

PRs em aberto e viés de recência. A taxa de merge é calculada sobre todos os PRs, mas uma fração relevante ainda não foi decidida (RQ1: Cursor e Copilot com ~20% em aberto). Como PRs recentes tendem a ficar em aberto, agentes com contribuições mais novas têm a aceitação subestimada. Mitigações: tornamos a fatia em aberto explícita (RQ1) e, idealmente, restringir a PRs decididos ou modelar o tempo até a decisão controlaria esse viés.

Amostragem da RQ5. A análise estática rodou sobre uma amostra aleatória de ~1,8 mil arquivos Python, com poucos do Claude Code (113). A amostra pode não representar o todo; rodar no dataset inteiro é trabalho pendente (§8).

Validade externa

O dataset cobre até agosto de 2025 e apenas projetos públicos do GitHub. Resultados podem não generalizar para código privado/corporativo ou para versões mais novas dos agentes. A RQ5, em especial, cobre apenas Python (linguagem com melhor suporte de linter), então seus achados não se estendem automaticamente a outras linguagens.

Validade de conclusão

A classificação de tipo de tarefa (pr_task_type) foi feita por um LLM e pode conter erros. Patches grandes não são fornecidos pela API do GitHub, limitando análises baseadas em diff. Na RQ5 isso é agravado porque analisamos apenas as linhas adicionadas do diff — um fragmento, não o arquivo completo após o merge —, o que pode subestimar ou superestimar problemas que dependem do contexto em volta.

8. Trabalhos futuros

  • Análise estática dos patches (RQ5 — já implementada de forma preliminar, ver §6.5): extrair os diffs de pr_commit_details e rodar linters por linguagem para medir densidade de problemas no próprio código, não só na aceitação. Próximos passos: estender para além de Python, rodar no dataset inteiro (não só na amostra) e adicionar análise de segurança (bandit/CodeQL) e o maintainability rating do SonarQube numa amostra de repos reconstruídos, para triangular.
  • Testes dinâmicos (RQ6 — amostra já selecionada, ver §6.6): rodar o runner (src/mutation_coverage.py) que reconstrói os repos e mede cobertura (pytest-cov) e mutation score (mutmut) do código dos agentes. Próximos passos: executar o runner na amostra, ampliar o tamanho, e explorar test smells (ex.: testes super-mockados) nos testes que os agentes escrevem.
  • Métricas de tamanho e complexidade: relacionar linhas adicionadas/removidas e número de arquivos com a chance de aceitação.
  • Tempo de turnaround: medir merged_at − created_at como proxy de velocidade/atrito.
  • Modelagem estatística: regressão logística da probabilidade de merge controlando por agente, tipo de tarefa, tamanho e popularidade do repositório.
  • Análise qualitativa: amostrar comentários de revisão e categorizar os tipos de problema apontados (bug, estilo, segurança…).

9. Reprodutibilidade

9.1 Stack tecnológica e por que Python

Toda a análise foi implementada em Python. A escolha não é arbitrária: ela acompanha o formato dos dados, as ferramentas da comunidade de mineração de repositórios (MSR) e as extensões previstas em §8.

FerramentaPapel na pesquisa
pandasLer as tabelas .parquet, fazer os join por pr_id/repo_id, derivar merged/rejected e agregar os proxies por agente e tipo de tarefa (§5).
pyarrowBackend de leitura/escrita do formato colunar Parquet usado pelo dataset.
huggingface_hubAcesso direto às tabelas via hf://datasets/hao-li/AIDev, sem download manual.
matplotlibGeração dos gráficos PNG das métricas (ex.: taxa de merge, CHANGES_REQUESTED).
ruffRQ5 — linter rápido (Rust) que conta warnings de estilo/smells sobre os patches.
lizard / radonRQ5 — complexidade ciclomática e maintainability index do código adicionado.
banditRQ5 (futuro) — varredura de padrões inseguros (CWE) em código Python.
pytest / pytest-covRQ6 — executa a suíte de testes do repo e mede a cobertura das linhas adicionadas.
mutmutRQ6 — teste de mutação: injeta defeitos no código do agente e mede o mutation score.

A escolha de uma stack de bibliotecas pip para a RQ5 (em vez do SonarQube) é deliberada: ferramentas como ruff/lizard rodam como biblioteca/CLI, são scriptáveis e reproduzíveis sobre centenas de milhares de diffs isolados. O SonarQube exige servidor e projetos compiláveis inteiros — inviável em lote e difícil de reproduzir para a banca —, ficando reservado a uma validação qualitativa em amostra (§8).

As razões da escolha, ligadas diretamente a este estudo:

  • O dataset é .parquet e o ecossistema Python (pandas/pyarrow) é o padrão de fato para ler, juntar e agregar essas tabelas normalizadas (§4) com poucas linhas.
  • Acesso nativo ao Hugging Face: a leitura via hf:// é Python-first, o que simplifica a coleta.
  • MSR é majoritariamente em Python: como o estudo se insere no Mining Challenge do MSR 2026, usar a linguagem padrão da comunidade torna o trabalho mais fácil de reproduzir e revisar.
  • Pipeline único, análise → gráfico: agregação (pandas) e visualização (matplotlib) ficam no mesmo script, sem troca de ferramenta.
  • Extensibilidade: os trabalhos futuros (§8) — análise estática de diffs, regressão logística (scikit-learn/statsmodels), classificação por LLM — têm bibliotecas Python maduras, evitando reescrever o pipeline.

9.2 Como reproduzir

Todo o código está versionado em github.com/RuthMaria/aidev-analysis. Para reproduzir os resultados:

# 1. Instalar dependências
pip install -r requirements.txt

# 2. Explorar as tabelas do dataset
python src/explore.py

# 3. Gerar as métricas de qualidade RQ1-RQ4 (tabelas + gráficos PNG)
python src/quality_analysis.py

# 4. Rodar a análise estática RQ5 (amostra de arquivos Python)
python src/static_analysis.py

# 5. RQ6: selecionar a amostra de PRs para os testes dinâmicos
python src/sample_for_testing.py            # gera outputs/test_sample.csv
# (passo seguinte: rodar o runner de cobertura+mutação sobre essa amostra)

Por padrão os scripts leem direto do Hugging Face. Para usar uma cópia local dos .parquet, defina a variável de ambiente AIDEV_DATA: 

# PowerShell (Windows)
$env:AIDEV_DATA = "C:\Users\Ruth\Downloads\aidev"
python src/quality_analysis.py

A RQ5 analisa por padrão uma amostra de arquivos .py (AIDEV_RQ5_SAMPLE, padrão 5000) para ser rápida; use AIDEV_RQ5_SAMPLE=0 para rodar no dataset inteiro.
